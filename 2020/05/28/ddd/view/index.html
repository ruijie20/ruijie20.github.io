DDD概要

发表于 2020-05-28

DDD

DDD是一套综合软件系统分析和设计的面向对象建模方法，通过DDD提高架构和代码的可理解性，从而降低了软件的维护成本。统一语言很重要。
DDD分段式协作的三个阶段：

  • 战略设计：澄清业务与问题
  • 战术设计：建立抽象模型
  • 技术实现：深入实现细节
    DDD 适合于中大型组织的长期项目——因为拥有清晰、完整的业务规则

一些概念

事件风暴

主要解决的问题是澄清业务问题，为构建领域模型提供输入。

领域事件
  • 领域事件会在系统中留下痕迹—-数据的变化
  • 领域事件具有原子性，必须拆到不可拆分为止
  • 事件流很大程度上是业务逻辑，挑战的是业务逻辑是否严丝合缝
  • 事件流必须忽略技术实现细节
限界上下文

一个显示边界，领域模型便存在于这个边界之内。在边界内，通用语言中的所有术语和词组都有特定的含义，模型需要准确反应通用语言。最好的情况子域应该一一对应一个限界上下文。

在项目中，其可以对应单体中的一个模块，或者是微服务中的一个服务。限界上下文意味着对领域模型、团队合作以及技术风险的控制。限界上下文是可控的，意味着对系统架构和组织结构是可控的。

  • 领域逻辑层面：确定了领域模型的业务边界，维护了模型的完整性与一致性，从而降低系统的业务复杂度。
  • 团队合作层面：确定了开发团队的工作边界，建立了团队之间的合作模式，避免团队之间的沟通变得混乱，从而降低系统的管理复杂度。
  • 技术实现层面：限界上下文确定了系统架构的应用边界，保证了系统层和上下文领域层各自的一致性，建立了上下文之间的集成方式，从而降低系统的技术复杂度
    划分限界上下文的方法：

  1. 聚合分组法
    聚合汇聚 -> 聚合拆分
    通过三因素进行调整

    • 业务因素：业务变化速率，业务承载量
    • 团队因素：组织结构映射
    • 技术因素：高并发，伸缩性，安全性等

  2. 关键事件法

上下文映射图

表示两个或多个限界上下文的映射关系。作用：

  • 以帮助更直观的掌握全局，识别上下⽂的依赖关系，为安排开发计划提供参考
  • 逆向检查，上下文划分是否合理，会否出现双向依赖，环形依赖等

构造块

  1. 实体
  2. 值对象
  3. 工厂
  4. 聚合
    由实体和值对象在一致性边界内组成。原则：
    • 在一致性边界内建模真正的不变条件。对于一个设计良好的限界上下文，能保证在一个事务中只修改一个聚合实例
    • 小聚合
    • 通过唯一标识引用其他聚合
    • 在边界之外使用最终一致性

聚合根有独立的生命周期，实体的生命周期从属于其所属的聚合，聚合根没了，实体也就没了，实体完全由其所属的聚合根负责管理维护

架构

  • 六边形架构
  • CQRS
  • 事件驱动架构
